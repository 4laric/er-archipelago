# SPEC: Gen-input hash gate — make "NEEDS WINDOWS REGEN" structurally impossible

**Status:** draft (2026-07-10) · **Owner:** Alaric · **Scope:** `gen_data.py` output headers, `build.ps1`,
`gen-greenfield.ps1`, CI · **Motivating diagnosis:** the two-machine split (Linux sandbox derives + tests;
Windows packages whatever generated bytes are *resident*) admits a bug class no test can see. Green CI and a
broken package are simultaneously possible, because tests never see the shipped bytes. The 10 "NEEDS WINDOWS
REGEN" markers are instances; the cp1252 empty-catalog incident and mount truncation are the same class in
other costumes. The marker convention treats this as a discipline problem. **It is a missing invariant.**

---

## 1. The bug class

`gen_data.py` reads inputs on machine A and writes 9 generated modules; `build.ps1 -Apworld` packages those
modules from disk on machine B. Nothing binds the packaged bytes to the inputs they were derived from. So:

- Edit `region_map.csv` in the sandbox, commit, don't regen on Windows → the package ships **stale** data.
  (This is exactly what the 10 markers are warning about, by hand.)
- A truncated write (mount truncation gutted 10 files in one session, 2026-07-10) → a generated module ships
  **partial** and every test that imports a still-resident older copy stays green.
- An encoding fault (cp1252 → empty catalog) → generator "succeeds", ships an **empty** table, tests that ran
  pre-fault stay green.

All three are the same shape: **shipped bytes ≠ f(current inputs)**, undetectable by tests because tests
validate freshly-derived data, never the resident/packaged artifact.

## 2. Definition of done

`gen_data` embeds a content hash of **all its inputs** into **every** generated file. Any consumer — the
installer, the packager, CI — recomputes that hash from disk and **refuses to proceed on mismatch**. The
"NEEDS WINDOWS REGEN" comment convention is deleted, because the build now enforces what the comment asked a
human to remember.

---

## 3. Design

### 3.1 Input manifest

The hash must cover *everything the output depends on*, tracked or not (the artifacts are filter-repo'd out,
so the hash is the **only cross-machine witness** that both machines saw identical inputs):

- **Tracked:** `greenfield/region_map.csv`, `greenfield/region_overrides.tsv` (from the provenance spec),
  `greenfield/item_tiers.tsv`, and **`gen_data.py` itself** — the transform is part of the input; a code
  change with unchanged data must invalidate the stamp.
- **Untracked artifacts** (`elden_ring_artifacts/`): `grace_flags.tsv`, `grace_region_map_*.tsv`, `event/`
  (EMEVD dir — hash the concatenation of member files, sorted), `msg/item*-msgbnd-dcx`, `vanilla_er/` params.
- **Datamine intermediates** consumed by gen_data: `eldenring_gf/boss_drops.py`, `eldenring_gf/boss_healthbars.py`
  (themselves generated by the datamine scripts run earlier in `gen-greenfield.ps1`). Include their bytes so a
  stale datamine step is caught too.

`inputs_hash = sha256( sorted list of (logical_path, sha256(bytes)) )`. Directory inputs hash their member
`(relpath, sha256)` pairs sorted, so ordering is stable across OSes. A shared `tools/gen_manifest.py` computes
this and is imported by both `gen_data.py` and the PowerShell gate (invoked via `python -m tools.gen_manifest`)
so **one implementation** defines the hash on both machines.

### 3.2 Stamp written into every output

`gen_data` computes `inputs_hash` once at start and writes a uniform header into each of the 9 modules:

```python
_GEN_STAMP = {
    "inputs_hash": "sha256:ab12…",
    "gen_data_sha": "sha256:…",     # gen_data.py's own hash (redundant w/ manifest, but human-legible)
    "generated_utc": "2026-07-10T…",
    "host": "windows|linux",
    "rows": 5164,                    # per-module cardinality (see 3.4)
}
```

Plus a sibling `greenfield/eldenring_gf/_gen_stamp.json` carrying the same `inputs_hash` and the per-module row
counts, for cheap tooling access without importing Python modules.

### 3.3 Gates

Three enforcement points, cheapest to strongest:

1. **Installer/packager gate (the one that retires the markers).** In `gen-greenfield.ps1` (post-gen, pre-copy)
   and `build.ps1 -Apworld` (pre-zip): recompute `inputs_hash` from disk via `python -m tools.gen_manifest`,
   compare to `data.py`'s `_GEN_STAMP["inputs_hash"]`. Mismatch → `throw` with the offending input listed.
   A data edit without a regen now **fails the build** instead of shipping stale — the marker becomes
   unnecessary because the machine remembers.
2. **All-stamps-equal test** (tier-A, runs in the sandbox): assert the `inputs_hash` in all 9 modules +
   `_gen_stamp.json` are **identical**. A partial regen (some modules new, some stale — the truncation /
   crash-mid-run signature) yields divergent stamps → red. Catches truncation as a *bonus* without relying on
   the pre-commit size gate.
3. **CI freshness check** (`run_ci.ps1`, `ci-linux.sh`): recompute `inputs_hash` from the checked-out tree and
   assert it equals the committed stamp. A commit whose generated data lags its inputs fails CI on **either**
   machine — this is what makes a stale commit un-mergeable, closing the "generated-data-can-lag" gotcha.

### 3.4 Cardinality / non-emptiness in the stamp

Embed each module's row/entry count (`rows`) in its stamp and assert `rows > 0` (and within a sane band) in
the all-stamps test. This catches the **empty-catalog** class directly: a cp1252/encoding fault that produces
an empty table changes `rows` to 0 → red, even if `inputs_hash` happened to match. (Pair with the existing
utf-8 + `ascii()`-keys rule, don't replace it.)

## 4. What this deletes

- The **"NEEDS WINDOWS REGEN" comment convention** — the build gate enforces it. (Grep the tree and strip the
  markers once gate #1 lands.)
- The class of "green CI, broken release" — impossible once shipped bytes are bound to input hash.
- Reliance on the pre-commit **truncation** gate as the *only* net — the all-stamps + cardinality checks catch
  truncation and emptiness structurally.

## 5. Migration path

1. `tools/gen_manifest.py` — the shared hash function (used by Python + PS). ~1 hr.
2. `gen_data.py` — compute `inputs_hash`, write `_GEN_STAMP` into all 9 outputs + `_gen_stamp.json`.
3. Gate #1 in `gen-greenfield.ps1` + `build.ps1 -Apworld` (the release-critical one — land first).
4. All-stamps-equal + cardinality test into the tier-A set.
5. CI freshness check; then strip the "NEEDS WINDOWS REGEN" markers.

## 6. Files touched

- `tools/gen_manifest.py` — **new**, shared hash (Python entrypoint callable from PS).
- `greenfield/gen_data.py` — compute + emit `_GEN_STAMP` in every `open(...,'w')` block (9 sites:
  data.py:1031, region_open_flags:1148, boss_data:1200, region_graces:1270, shop_data:1343,
  missable_locations:1378, item_ids:1543, item_tiers:1670, location_tags:1780, boss_sweeps:1897) +
  `_gen_stamp.json`.
- `greenfield/gen-greenfield.ps1` — recompute-and-compare after gen_data, before install/copy.
- `build.ps1` — same gate in the `-Apworld` packaging block (before the zip at ~:160).
- `greenfield/eldenring_gf/tests/test_gf_gen_stamp.py` — **new**, all-stamps-equal + cardinality.
- `run_ci.ps1`, `greenfield/ci-linux.sh` — freshness assertion.

## IMPLEMENTATION NOTES (2026-07-10, landed)

**Shipped and verified in-sandbox:**
- `tools/gen_manifest.py` -- the one hash definition. Newline-normalizes text inputs (CRLF/LF
  invariant across Windows/Linux checkouts), hashes only the files gen_data actually reads (region_map
  + item_tiers + region_overrides + the two ItemLot-adjacent shop csvs + grace tables + the EMEVD
  `event/` dir + the 15 item-name FMG xml + boss_drops/boss_healthbars + gen_data.py itself). 1038
  inputs, deterministic across runs. It **found a real bug on first run**: `item_tiers.tsv` is at repo
  root, not `greenfield/` (fixed).
- `gen_data.py` -- appends a uniform `_GEN_STAMP` (inputs_hash, utc, host, module, body_sha256) to all
  10 generated modules + writes `_gen_stamp.json` with per-module body checksums and semantic counts.
- `greenfield/eldenring_gf/tests/test_gf_gen_stamp.py` -- Predicate A (one inputs_hash across all
  modules), B (body checksums intact -> truncation/partial-regen), C (count floors -> empty-catalog),
  D (freshness vs disk; skips where artifacts absent). 4 pass / 2 skip in the installed-world harness.
- `gen-greenfield.ps1` -- gate wired after `gen_data.py`: `python tools/gen_manifest.py --verify
  eldenring_gf/_gen_stamp.json`, throws on stale/unverifiable. Proven: OK when fresh, **STALE (exit 3)**
  when an input changes, OK on restore.

**Not yet done:** build.ps1 `-Apworld` packaging gate (confirm which world release packages first);
CI freshness assertion in `run_ci.ps1` / `ci-linux.sh`; stripping the "NEEDS WINDOWS REGEN" markers
(do this only after the gate has run green through one full Windows build).

**Note on committing generated data:** the sandbox regen drifts from the committed modules (7
pre-existing `test_gf_dlc` / boss-tag failures reproduce on committed data too -- environmental, not
from this work). So the stamped modules + `_gen_stamp.json` must be regenerated on **Windows**
(`build.ps1 -Greenfield`) before commit; do not commit the sandbox-regenerated data.
