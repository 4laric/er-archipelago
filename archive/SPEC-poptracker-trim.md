# SPEC: PopTracker pack — trimmed/lean pool awareness (M1.5)

Status: SPEC, not started (Alaric, 2026-06-15). Slot_data half already landed (see Contract below).
Builds on: `poptracker/` M1 (SPEC-poptracker-pack.md, BRIEF-poptracker-pipeline.md), the
`location_pool` option (options.py `LocationPool`), and the trim semantics in SPEC-check-trim.md
(`__init__.py:_in_location_pool`).

## Problem

The pack is generated from the FULL `locations.py` — every check, i.e. the `location_pool = all`
footprint (~4,906 tracked loc-ids). When a seed is rolled with `location_pool: trimmed` (~2,150) or
`lean` (~500), the apworld never CREATES the dropped checks, so they are never sent and never clear.
On the tracker they sit forever as phantom uncleared sections: wrong totals, misleading
"checks remaining", and reachability counts that never reach zero. The pack has no concept of the
pool at all (no `trim` reference anywhere in `poptracker/`).

## Goal

When connected to a trimmed or lean seed, the pack shows only the checks that seed actually contains.
Checks the seed dropped are not counted as outstanding. Connecting to an `all` seed is unchanged.

## Contract (CHANGED — read this)

The M1 brief promised "Contract: none — read slot_data only, never require a new key." **That promise
is now void.** As of 2026-06-15 the apworld ships `"location_pool"` in `fill_slot_data`'s `options`
block (added after `world_logic`). The pack now DEPENDS on that key.

- Back-compat is mandatory: if `slot_data.options.location_pool` is absent or nil, treat it as `0`
  (all) — i.e. an older server/seed shows every check, exactly today's behaviour. Never error on a
  missing key.
- Version lockstep: bump `manifest.package_version` in step with the apworld `versions` string when
  this lands, since pool membership must match the seed's apworld build.

## Design

Two halves, mirroring how `world_logic`/`enable_dlc` are already handled.

### A. Generator — emit per-location pool membership

Single source of truth must stay the apworld. Do NOT re-derive the trim rules in Python inside the
generator (they will drift from `_in_location_pool`).

1. **Refactor the apworld predicate to be import-callable (prerequisite).** Today
   `_in_location_pool`, `_is_bad_and_remote`, and `_remoteness` are `World` methods, but their trim
   decision reads only `data` + module-level state (`item_table`, `JUNK_GOODS`, `ITEM_TIERS`,
   `LOC_GRACE_DIST`, the `GRACE_*_M`/`REMOTE_THRESHOLD` constants) — not `self.random` or per-seed
   state. Extract the pure core to a module-level function:

   ```python
   def location_in_pool(data: ERLocationData, pool: int) -> bool: ...
   ```

   and have the three methods delegate to it. This is a safe, test-coverable refactor that also lets
   `trim_report.py` drop its duplicated rule copy. (If any helper turns out to need `self`, pass the
   needed value in explicitly — keep the function pure.)

2. **Generator imports and calls it.** `gen_poptracker.py` already runs in RUNTIME mode (imports
   `locations.py`/`items.py` with `BaseClasses` stubbed). Import `location_in_pool` the same way and,
   for each location, compute membership at pool 1 and pool 2 (pool 0 is always true).

3. **Emit a generated Lua table** `scripts/loc_pool.lua` next to `loc_map.lua`:

   ```lua
   -- generated by tools/gen_poptracker.py — AP location id -> retained-pool bitmask
   -- bit0 (1) = kept in trimmed, bit1 (2) = kept in lean; all-pool keeps everything
   AP_LOC_POOL = { [12345]=3, [12346]=1, [12347]=0, ... }
   ```

   Pools are NOT strictly nested (trimmed drops bad-and-remote gear & junk-goods that lean may keep
   via its `church`/`basin`/`fragment`/… tags, and vice-versa), so a bitmask of explicit per-pool
   membership is required — a single "min pool" integer is wrong.

4. **Extend the generator's stdout line** with `pool_kept_trimmed` / `pool_kept_lean` counts (next to
   the existing `locations`/`location_ids` fields) so a regen visibly reports the split, and fold
   `loc_pool.lua` into the `--check` staleness gate.

### B. Autotracking / logic — hide dropped checks on connect

1. `init.lua`: `LoadScript("scripts/loc_pool.lua")` (before `autotracking.lua`, after `loc_map.lua`).
2. `autotracking.lua:on_clear` already reads `slot_data.options`; add
   `WORLD_POOL = opt["location_pool"] or 0` and log it alongside `world_logic`/`enable_dlc`.
3. Mechanism for hiding — preferred then fallback (the load-and-validate pass decides which survives
   contact with PopTracker):
   - **Preferred — native visibility rule.** Give each generated location/section a `visibility_rule`
     that calls a Lua helper `pool_has(loc_id)` → `WORLD_POOL == 0 or bit_set(AP_LOC_POOL[loc_id], WORLD_POOL)`.
     Dropped sections then don't render and don't count toward totals. Cleanest; verify PopTracker
     honours per-section visibility rules in the list-only layout.
   - **Fallback — auto-clear on connect.** If runtime visibility proves fiddly, in `on_clear` walk
     the pack's location ids and for any not in `WORLD_POOL`, set its section
     `AvailableChestCount = 0` (the same call `on_location` uses). Dropped checks then read as
     done rather than outstanding — totals are correct, though they show as "checked" not "absent".
     Acceptable v1.
4. Reachability interaction: `logic.lua` counts are region-level and don't enumerate locations, so no
   change needed there — but if a future per-region "X/Y checks" readout is added, it must count only
   pool-kept sections.

## Test plan

- Generator: after the refactor, `python3 tools/gen_poptracker.py` emits `loc_pool.lua`; its
  trimmed/lean kept-counts match `tools/trim_report.py` (which now calls the same `location_in_pool`).
  Expected order-of-magnitude: all ≈ 4,906, trimmed ≈ all − 392 (24 gear + 330 junk + 38 somber),
  lean ≈ 500. `--check` exits 1 if `loc_pool.lua` is stale.
- In PopTracker: connect to an `all` seed → every section shows (unchanged). Connect to a `trimmed`
  seed → the ~392 dropped sections disappear (or auto-clear); the rest track normally. Connect to a
  `lean` seed → only meaningful/boss/key/progression sections remain. Disconnect/reconnect to a
  different-pool slot updates the visible set. Older seed with no `location_pool` in slot_data → shows
  all checks, no error.

## Out of scope

- Map/pin display of pool state (M3, blocked on map-art licensing).
- Per-location item RULES (M2 apworld declarative-rules refactor) — independent of this.
- Changing what the apworld trims; this spec only mirrors existing `_in_location_pool` decisions.
